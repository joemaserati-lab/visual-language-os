System / Components

Components translate rules into interface decisions.

The component layer keeps visual language consistent without turning the system into a template. Each module has a purpose, usage boundary and visible proof.

Component Laws

Reusable does not mean generic.

Components should make decisions repeatable: hierarchy, density, emphasis and action all stay traceable to the same operating rules.

C-001Start with purpose.

Choose a component because it clarifies a job: action, status, grouping, evidence, navigation or conversion.

C-002Reuse the smallest stable pattern.

Do not invent a new block when an existing card, badge, rule or table already expresses the decision.

C-003Variants must reduce ambiguity.

Default, compact, emphasis and dark states should signal context, not personal preference.

Button

Button

Purpose. Trigger one clear action, especially when the user is expected to commit, continue or contact.

Use rules. Keep labels short and verb-led. Use one emphasis button per decision area. Let spacing make the action visible before color does.

Avoid when. The target is pure navigation, repeated metadata or a secondary text reference.

Variants. Default, compact, emphasis, dark.

Badge

Badge

Purpose. Label status, category or system role without competing with headings.

Use rules. Keep text uppercase and brief. Place before a heading, inside a specimen or as a small state marker.

Avoid when. The label becomes the primary message or carries complex instructions.

Card

Card

Purpose. Group one idea, rule or destination into a contained reading unit.

Use rules. Use a clear eyebrow, title and short explanation. Cards should compare well in a grid and not depend on decoration.

Avoid when. Content needs sequential reading, dense tabular comparison or a primary narrative flow.

Variants. Default, compact, emphasis, dark.

Rule Block

Rule block

Purpose. Codify a decision as a named law, checklist item or operating principle.

Use rules. Pair a compact identifier with a decisive title and one explanatory paragraph.

Avoid when. The content is promotional, optional commentary or a long essay.

Variants. Default, compact, emphasis, dark.

Token Table

Token table

Purpose. Compare tokens, timings or named values in a scannable structure.

Use rules. Keep columns predictable: token, role and usage. Make values more prominent than descriptions.

Avoid when. The content is narrative or each row needs a different structure.

Variants. Default, compact, emphasis, dark.

Avoid when. The item is a small utility link, generic list entry or text-only reference.

CTA Block

CTA block

Purpose. Close a page section with one focused action and a concise reason to act.

Use rules. Use one statement, one supporting paragraph and one action. Keep competing links out of the block.

Avoid when. The user still needs comparison, education or multiple decision paths.

Variants. Default, compact, emphasis, dark.

Navigation Item

Navigation item

Purpose. Move users between stable sections while showing the current location.

Use rules. Keep labels noun-based, short and consistent. Active state should be clear without needing animation.

Avoid when. The destination is temporary, promotional or better represented as an action button.
