System / Components

Components translate rules into interface decisions.

The component layer keeps visual language consistent without turning the system into a template. Each module has a purpose, usage boundary and visible proof.

Component Laws

Reusable does not mean generic.

Components should make decisions repeatable: hierarchy, density, emphasis and action all stay traceable to the same operating rules.

C-001Start with purpose.

Choose a component because it clarifies a job: action, status, grouping, evidence, navigation or conversion.

C-002Reuse the smallest stable pattern.

Do not invent a new block when an existing card, badge, rule or table already expresses the decision.

C-003Variants must reduce ambiguity.

Default, compact, emphasis and dark states should signal context, not personal preference.
