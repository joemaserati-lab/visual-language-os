System / Accessibility

Access is part of the visual language.

Every accessibility rule must be visible, testable and resilient when motion, fonts, color, pointer input or perfect reading conditions are unavailable.

Operating principle

Readable before expressive.

A-000The fallback state is the design.

Accessibility is not a remediation layer. The page must still communicate hierarchy, state and action when the most expressive visual layer is removed.

A-000Do not encode meaning once.

Use text, structure, shape, spacing and symbols together so no single sensory cue carries the whole message.

A-001

Reduced motion

Motion may reveal sequence, but it must never be required to read content. Respect prefers-reduced-motion, keep the final state available immediately and avoid large parallax or vestibular movement for essential information.

Correct ✓ Stable state
Step 2 is visibleLabel + number explain progress.

The message is readable without waiting for animation.

Incorrect ✕ Motion-only reveal
Hidden until spinMeaning depends on movement.

The user must tolerate motion to discover the state.

  • Reduced-motion preference disables non-essential transitions.
  • Content is present in the DOM before animation starts.
  • Animated state changes also include text labels or fixed indicators.
A-002

Font fallback

Brand fonts can fail or load late. Fallback stacks must preserve size, rhythm and legibility so the interface remains stable with system fonts.

Correct ✓ Fallback stack
System-ready headlineRedaction → Geist → system-ui → sans-serif

The same hierarchy survives custom font failure.

Incorrect ✕ Fragile font
Compressed identitySingle decorative face, no usable fallback.

The design depends on one file to remain legible.

  • Every font family includes readable fallbacks.
  • font-display: swap or equivalent avoids invisible text.
  • Line-height and spacing work with both brand and system fonts.
A-003

Color contrast

Text, icons and component states need enough luminance contrast against their backgrounds. Contrast supports fatigue-free reading and must be checked in normal, hover, focus and disabled states.

Correct ✓ 12.7:1
Readable labelPASS badge + underline pattern

High contrast is reinforced by a written PASS marker.

Incorrect ✕ 2.1:1
Faint labelFAIL badge + dotted pattern

Low contrast is identified by text and pattern, not color alone.

  • Body text and UI labels meet expected contrast thresholds.
  • Critical states are checked on their real backgrounds.
  • Disabled styles remain distinguishable without becoming unreadable.
A-004

Link recognition

Links must look interactive in running text. Do not rely on color alone; use underlines, icons, weight, spacing or explicit action copy.

Correct ✓ Text + underline

Read the accessibility checklist ↗ before release.

The underline and arrow identify the link without color.

Incorrect ✕ Color-only

Read the accessibility checklist before release.

The colored words have no persistent affordance or semantic link.

  • Inline links are underlined or otherwise visibly marked.
  • Link text describes destination or action.
  • Hover and focus states add cues beyond color.
A-005

Semantic heading structure

Headings are navigation landmarks for assistive technology and scanning readers. Use one logical h1, then ordered sections without skipping levels for visual size.

Correct ✓ Ordered outline
  1. h1 Page title
  2. h2 System rule
  3. h3 Checklist detail

Visual scale follows semantic order.

Incorrect ✕ Skipped levels
  1. h1 Page title
  2. h4 Big-looking section
  3. h2 Nested detail

Screen-reader navigation becomes unpredictable.

  • There is one primary page heading.
  • Heading levels increase by one level at a time.
  • Do not choose heading tags for visual size only.
A-006

Keyboard focus

Every interactive element must be reachable and visibly focused with a keyboard. Focus indicators need shape, thickness and offset so they are visible in light and dark contexts.

Correct ✓ Visible focus

The outline, offset and key symbol show current position.

Incorrect ✕ Hidden focus

The button gives no keyboard location cue.

  • Tab order follows the visual reading order.
  • Focus is never removed without a stronger replacement.
  • Custom controls support Enter, Space and expected arrow-key behavior.
A-007

Text readability

Readable text uses comfortable line length, line-height, spacing and plain language. Editorial tone should not force users to decode dense blocks or tiny type.

Correct ✓ Comfortable measure

Short paragraphs, clear labels and generous line-height make the page easier to scan, translate and revisit.

Structure reduces cognitive load.

Incorrect ✕ Dense block

Long uninterrupted paragraphs with compressed spacing and wide measures make it difficult to retain place, compare options or recover after interruption.

Density hides meaning.

  • Body copy stays near 45–75 characters per line.
  • Line-height supports scanning and zoom.
  • Paragraphs, lists and labels are written in direct language.
A-008

Non-reliance on color

Color can support status, priority and grouping, but it cannot be the only signal. Pair color with text, icons, patterns, borders or placement.

Correct ✓ Multi-signal status
✓ ApprovedSolid border + PASS label + check icon

Status remains clear in grayscale.

Incorrect ✕ Color-only status
StatusMeaning is implied only by hue.

Users who cannot distinguish the hue lose the message.

  • Status indicators include text or symbols.
  • Charts and diagrams use labels, patterns or direct annotation.
  • Error and success states are understandable in grayscale.
